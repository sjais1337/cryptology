# Matching 
There are two files for match checking, one of them is CPU based the other is GPU based. The CPU based version is used for multiple iterations, and uses some degree of randomness. The GPU based version is however deterministic, and is used to find the max bits matching.

Since the GPU version is deterministic, its not feasible to run multiple iterations of it, however it yields a match for the first 32 bits in around 4-5 seconds. If we leave it running for longer, we might potentially be able to reach 40-48 bits in practical time. We also found a 40bit match, but it has a whitespace in the beginning, " LKe:S{@iitk.ac.in" has the hash "66c58d5e90080fe87329250716b6463833638204c128a32e9105fa231a02ad75", which matches the first 40 bytes with "shivanshj24@iitk.ac.in" hash, i.e. "66c58d5e901b8c222af0233bef977dc94cf32239bbcc9649c816858eba221e98".

However, in the CPU version, we obtained the following as the Average Iterations Taken: 26682157.64. Taking the logarithm base 2, we obtain log_2(26682157.64) = 24.66. If the number of bits we were trying to match was n, then the probability of finding a match with a single random input is 1/2^n. Thus, it should take approximately 2^n iterations to get a n bit match. Here, the value obtained is close to 2^24, that is why we are getting the specified value. 

To control the number of matching bits, modify the "NUM_BITS_TO_MATCH" macro in the definitions, present on line 10 for match_check.cu and line 13 for match_check.c.
